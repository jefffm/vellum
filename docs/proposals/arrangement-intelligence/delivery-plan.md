# Tracer-Bullet Delivery and Acceptance

Status: Draft subordinate proposal

This document intentionally contains no effort or time estimates.

## Operational prerequisites

Before arrangement-intelligence work depends on the browser surface:

- bind the local API to loopback and restrict CORS;
- establish the compiled-SVG sanitization boundary;
- execute every externally influenced notation compiler in a pinned, disposable, no-network, resource-bounded host-isolated sandbox that exposes no secrets or writable host path and fails closed when unavailable;
- normalize typed safe API errors;
- add coordinated frontend and API development startup;
- repair completed-artifact handoff;
- make review correction failure recoverable; and
- add one real-browser fixture workflow.

## First tracer

The first tracer is deliberately narrow:

```text
one immutable source
→ one reviewed Source Truth snapshot
→ one Analysis Record
→ one minimal Arrangement Plan
→ one Performance Brief
→ one Target Configuration
→ one Arrangement Search and selected score
→ one Evaluation Case and Card
→ one exact baseline comparison
→ one workbench adoption
```

Everything outside that path may remain explicitly unknown or not evaluated. The tracer does not require global learning, multiple plan alternatives, cross-domain generality, live model evaluation, or a complete fixture corpus.

### First-tracer readiness

- terminology required by the tracer is resolved;
- end-to-end Evaluation Case begins from source and briefs;
- resolved Evaluation Manifest pins all transitive inputs;
- current known output is preserved as an explicit baseline with known defects;
- one proportional Plan kind is supported;
- intended performer, difficulty, tempo context, and reliability goal are declared;
- target instrument representation is exact enough for claims made;
- absolute results and comparative deltas are separate;
- hard gates and unknown dimensions are explicit;
- one mutation proves the principal evaluator catches a known defect;
- artifact handoff and versioned adoption work in a real browser; and
- the report renders safely.

## Delivery sequence after the tracer

1. Expand Source Truth consequence and recovery coverage.
2. Add the remaining proportional Arrangement Plan kinds as fixtures require them.
3. Complete multi-string Instrument Instance and capability composition.
4. Replace proxy ranking with observable dimensions and honest comparison.
5. Add phrase-level target realization for the initial target.
6. Add Workbench playtest and reviewer-role evidence.
7. Add calibration candidates with strict dataset isolation.
8. Complete named baroque-guitar, lute, and classical-guitar fixture corpora.
9. Reach initial-target parity across planning, realization, notation, playback, playtest, and evaluation.
10. Validate shared protocols with a materially different domain such as continuo realization.
11. Accept focused ADRs and migrate only components that benefit from the architecture.

## Initial-target parity

Parity requires all three priority targets to support:

- purpose-scoped Source Truth;
- compatible Arrangement and Performance Briefs;
- proportional Arrangement Plans;
- exact Instrument Instance Configurations;
- target-specific phrase realization;
- honest search outcomes;
- Preservation Audits and Transformation Reports;
- target-appropriate notation and canonical playback;
- score-anchored editing, alternatives, audition, and adoption;
- target-player or Owner Playtest evidence;
- Evaluation Runs, Cards, mutations, and baseline comparison; and
- explicit unevaluated dimensions.

The shared Greensleeves source supplies a cross-target baseline. Each target also needs a named legally reusable fixture that exposes its own mechanics and musical design. Lute requires representative stopped-course and diapason interaction. Classical guitar requires reviewed polyphonic voice and fingering invariants. Baroque guitar requires exact stringing, alfabeto, and the observed transition counterexample.

## Cross-domain validation

A non-fretted-position problem must reuse Source Truth, Planning, evaluation, lineage, and learning contracts without flattening its domain. Continuo realization is the leading case because it stresses generated voices, harmonic context, dissonance preparation, bass authority, and historical profiles.

Project-wide architecture is not accepted merely because one target works.

## Product acceptance

Every Golden workflow reports an Evaluation Card covering:

- source authority;
- plan realization;
- preservation and transformations;
- target mechanics and technique;
- history and analysis;
- notation and playback;
- workflow and recovery;
- human or physical evidence; and
- explicit Owner usefulness.

No overall grade hides a hard failure. A deterministic pass may coexist with failed usefulness; both remain visible.

## ADR boundaries

Focused ADRs should cover:

- Source Truth and iterative convergence;
- Performance Brief and Arrangement Plan lineage;
- Instrument Instance and course representation;
- serializable constraints and honest search outcomes;
- Evaluation Harness manifests, baselines, and promotion;
- Owner Playtest and calibration boundaries; and
- project-wide Arrangement Intelligence only after cross-domain validation.
