# Tech Debt Audit — Vellum

Generated: 2026-07-11

Adversarially reviewed: 2026-07-11

## Executive summary

- Vellum's domain architecture is coherent, but its current definition of **playable** is too weak to prevent physically awkward arrangements.
- The baroque-guitar path chooses fingerings greedily one event at a time; phrase-level search with explicit hand-transition costs is the most important musical-quality slice.
- The local API binds without an explicit loopback host, has no authorization boundary, and reflects arbitrary development origins. Close this small trust-boundary hole before undertaking the larger search project.
- Guided Start is a 3,218-line stateful application without an explicit persisted workflow model. Failed and partially completed workflows are not resumable.
- Candidate-ranking labels overstate their evidence: “voice leading,” “playability,” and “historical profile” are currently crude proxies.
- Harmony reduction preserves pitch-class presence rather than bass function, inversion, independent rhythm, or inner-voice behavior. Whether Faithful Reduction must preserve those accompaniment properties remains a product-policy decision.
- Workspace JSON is atomically written but not transactionally or concurrency-safe; records and manifests can diverge or lose concurrent updates.
- Browser tests often inspect source strings instead of executing behavior. The real OCR → review → arrangement → artifact → playback path is not covered by browser E2E in CI.
- `npm audit` reports 15 vulnerable packages, but audit severity is not product exploitability. Each affected dependency needs an exposure assessment before prioritization.
- Live testing exposed two important omissions from the original audit: a completed run can return to an empty artifact panel, and the two-process development setup can leave a hot frontend talking to stale compiled API code.

## Architectural mental model

Vellum is a local-first single-owner music workstation. The browser combines a Pi-backed chat surface with Guided Start and an artifact workbench. Express routes orchestrate source import, Audiveris OMR, score normalization, musicological analysis, candidate search, preservation auditing, engraving, playback, provider authorization, and versioned editing. Canonical state is persisted as immutable or versioned JSON records plus source and deliverable files inside Arrangement Workspaces; owner defaults and reviewed historical knowledge live separately.

The architecture intentionally separates Source Artifact, Score Transcription, Normalized Score, Analysis Record, Arrangement Candidate, Arrangement Score, Performance Interpretation, and Deliverable. That separation is real, not aspirational. The main weakness is that sophisticated lineage and product language sit on top of deliberately small musical heuristics and a browser workflow that has grown into a stateful application without an explicit workflow-state model.

## Evidence and severity method

Each finding records an evidence status so a code-reading hypothesis is not presented as equivalent to a reproduced failure:

- **Observed:** seen in a live user workflow.
- **Reproduced:** demonstrated by an automated or deterministic manual test.
- **Code-proven:** the reachable code path establishes the behavior, but no end-to-end reproduction was recorded.
- **Risk:** plausible from code inspection; exploitability or material impact still needs testing.
- **Capability gap:** absent functionality relative to the stated product direction.
- **Decision:** a product or policy choice that must be resolved before implementation can be called correct.

Severity describes impact, not scheduling. A product-blocking High may be the next musical milestone; Critical is reserved for catastrophic security, data-loss, or total-system failures under the documented deployment model.

## Recommended execution order

1. **Close the local trust boundary (F004/F025/F043/F044).** Bind loopback, restrict CORS, normalize safe errors, and establish the compiled-SVG boundary. These are smaller changes than phrase search and reduce exposure of every other API/resource risk.

2. **Fix development and artifact handoff failures (F041/F042).** Make the actual app reliably run and display completed work, then protect that flow with one real-browser E2E.

3. **Build phrase-level ergonomic search (F001–F003).** Introduce a `HandState` with fretted fingers, barre, position, held notes, and course assignments. Generate feasible successors per musical event, score transitions, and run a bounded beam over a phrase. Feed the resulting evidence into candidate ranking.

4. **Make Guided Start resumable (F017–F020).** Define persisted stages such as `source_saved`, `recognizing`, `transcription_review`, `analysis_review`, `target_search`, `projection`, and `complete`, each with exact record IDs.

5. **Replace misleading musical proxies (F005/F006/F008/F016).** Add bass function, independent rhythmic lineage, inversion, spacing, and voice motion where the selected policy requires them. Until then, label metrics honestly.

6. **Protect versioned storage (F026–F029).** Begin with per-workspace serialized mutation and revisions, add orphan recovery, and correct score lineage resolution. Do not introduce a database unless these simpler controls prove insufficient.

## Things that look bad but are actually fine

- `src/lib/instrument-model.ts:291` excludes open strings from fretted-span computation. That is correct: an open course does not extend the left-hand fretted span.
- `src/lib/transformation-report.ts:191` classifies pitch-class-preserving octave moves as `octave_relocated`, not `retained`. That explicit provenance distinction is desirable.
- `src/server/lib/subprocess.ts:84` launches configurable commands with `shell: false`; current callers provide controlled executables and arguments, so this is not shell injection by itself.
- `src/server/lib/workspace-store.ts:1020` builds paths from IDs, but strict ID validation plus filename sanitization and containment checks form a sound traversal boundary.
- `src/guided-start.ts:1188` prevents Escape from dismissing a critical Score-Anchored Review while retaining an explicit Cancel action. That is deliberate integrity protection, not an inaccessible trap by itself.
- Fixed-template `innerHTML` in `src/main.ts:157` is developer-owned static markup. It is distinct from the externally influenced compiled-SVG boundary in F043.
- The many versioned JSON files are not inherently a failed database design. They fit the local-first, inspectable prototype; the debt is missing concurrency and recovery semantics, not the use of files.

## Open decisions for the maintainer

- Is LAN/private-remote access intended, or should every server surface remain loopback-only until a remote-mode ADR exists?
- Should Faithful Reduction promise only Principal Voice, or should bass identity and selected inner-voice relationships become Preservation Targets for homophonic sources?
- Are ranking weights intended to make the winner authoritative, or merely explain an arranger-selected result?
- What score sizes and upload sizes must the UI and file store support before pagination, indexing, and streaming become mandatory?
- Should chat and Guided Start deliberately expose different input formats?
